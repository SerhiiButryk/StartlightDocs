## Lambda vs anonimus classes

Did you know lambda and anonimus classes have different runtime cost?

Let's dive deeper and find out why it happens. 

# Anonimus classes

Let's create a 'Test.kt' file with the source code:

```kotlin
fun interface Listener {
    fun call()
    fun message() = "Hello"
}

class A {
    private var listener: Listener? = null

    fun addListener(listener: Listener) {
        this.listener = listener
    }

    fun call() {
        listener?.message()
    }
}

fun main() {

    val a = A()

    // Creates an anonimus class
    a.addListener(object : Listener {
        override fun call() {
            println("Hello this is my message")
        }
    })

    a.call()

}
```

# Lambda in Kotlin

Now let's create a lambda. This is our new source code:

```kotlin
fun interface Listener {
    fun call()
    fun message() = "Hello"
}

class A {
    private var listener: Listener? = null

    fun addListener(listener: Listener) {
        this.listener = listener
    }

    fun call() {
        listener?.message()
    }
}

fun main() {

    val a = A()

    // Creates a lambda
    val listener = Listener {
        println("Hello this is my message")
    }

    a.addListener(listener)

    a.call()

}
```

# Summary

So, you can see that there is a difference. 

1. Anonimus class creates extra class file.

2. Lambda has some cost during creation, but no cost after that. Usually this is the preferred way. Compile might apply more optimization here.

## Lambda and local variabls

You might know you can access and modify local variable from lambdas. But how it actually works ? Let's find out.

First off all let's analyze this code:

```kotlin
fun main() {

    var clientErrors = 0

    val serverResponses = listOf("500 Internal Server Error", "404 Not Found")

    fun printProblemCount() {
        // Iterate through each http status response
        serverResponses.forEach { element ->
            if (element.startsWith("4")) {
                clientErrors++
            }
        }
    }
}
```

But there is a strange parameter - **Ref.IntRef**. What is it? Well actually this is our local variable.

Here is this class declaration:


Worth mentioning that there is also 'ObjectRef' class for accessing and modification of local object variables:

# Summary

Lambda accesses local variables using a wrapper object which stores a reference to local variable or variable itself if it's a primitive.

1. Compiler generates a wrapper object for every local variable which a lambda accesses.

2. Compiler can insert code of a lamda function if a function has inline modifier.

3. The second is good news as code can be a bit faster.
